# 模拟面试 Agent 讲解

本文档用于解释当前模拟面试功能如何实现，以及面试时可以怎样回答。

## 用户体验上发生了什么

1. 用户登录后上传简历。
2. 用户可以选择填写目标岗位、公司和岗位描述，也可以全部不填。
3. 后端解析文本型 PDF 或文本简历。
4. 如果用户填写岗位上下文，后端尝试联网检索公开岗位情报；没有配置 Key 或搜索失败时自动降级。
5. 系统将简历、JD 和岗位情报切成结构化 chunk，并召回 evidence。
6. 多 Agent 协同分析简历、岗位、证据和模型建议，生成至少 8 道主问题。
7. 用户回答后，评分服务给出分数、反馈和缺失点。
8. LiteFlow 规则链只判断是否需要追问，具体问题由追问生成器结合当前回答和简历上下文生成。
9. 如果回答不足，会产生 `Q1-F1`、`Q1-F2`、`Q1-F3` 形式的多轮追问。
10. 系统根据整体表现把题量保持在 8 道，或扩展到 12/15 道。
11. 完成后生成报告、能力雷达、问答回放和改进建议。

## 为什么要多 Agent

普通 LLM 调用只有一次 Prompt 和一次回答，很难解释中间判断过程。当前项目把面试拆成多个角色：

- 简历分析 Agent：提取项目、技能、经历和风险点。
- 岗位上下文 Agent：分析岗位要求、公司信息和技术匹配度。
- 出题设计 Agent：综合简历、岗位、evidence 和历史反馈生成问题。
- 回答评估 Agent：分析回答命中点、缺失点和追问方向。
- 面试总结 Agent：汇总整场表现。

这些角色不是简单换 Prompt，而是通过 `InterviewOrchestrator` 串联，并把中间步骤写入 MongoDB 的 `agent_run` 和 `agent_step_trace`。面试官问时可以说：我用多 Agent 做职责拆分和可追踪性，先保证链路可解释，再用 evaluation 去评估效果。

## 为什么追问要拆成规则裁决和问题生成

如果让大模型同时决定“要不要追问”和“追问什么”，输出容易不稳定。当前做法是：

- LiteFlow 规则链负责判断：分数是否低、是否缺失关键点、是否超过最大追问次数、状态是否允许追问。
- 追问生成器负责问具体问题：结合当前题目、用户回答、缺失点、简历技术实体和岗位上下文生成问题。

这样可以避免“请补充技术细节”这种泛问。模型输出不合格时，后端会回退到规则问题，例如围绕 Redis、JWT、RabbitMQ、MySQL、WebSocket 等已识别技术实体继续追问。

## 为什么题量不是固定 5 题

真实面试通常不会只问 5 个问题。当前策略是：

- 默认 8 道主问题起步。
- 如果回答质量中等但还需要补充覆盖面，会扩展到 12 道。
- 如果表现波动较大或证据不足，会扩展到 15 道。
- 追问不计入主问题数量，而是挂在父问题下。

前端会展示“已答主问题数 / 当前计划题数”，让用户知道题量会动态变化。

## 当前不足

- 扫描版 PDF 尚未 OCR。
- 正式评估集还需要继续扩充，不能提前写优化百分比。
- ASR/TTS 当前是降级闭环，不是真实语音识别和合成。
- 题目质量和追问质量仍需要更多真实简历/JD 样本校准。

## 面试回答模板

可以这样介绍项目：

> 我做的是一个基于 Spring Boot 3 和 Spring AI 的 AI 模拟面试 Agent 后端。项目不是单纯调用大模型，而是把简历解析、岗位情报、RAG evidence、多 Agent 出题、LiteFlow 追问裁决、动态题量、报告生成和 evaluation 评测串成完整闭环。用户可以上传简历，也可以输入目标公司和 JD；系统会基于真实简历内容和可选岗位上下文生成问题，并根据回答质量进行多轮追问和题量调整。

如果被追问“和普通 AI 对话有什么区别”，可以回答：

> 普通 AI 对话没有明确状态、证据、评分和追问策略。本项目每轮面试都有会话状态、题目快照、Agent trace、RAG evidence、评分结果和追问链，数据分别落到 MySQL、MongoDB 和 Redis。这样系统可以恢复、回放、评估，也能用测试集量化改进效果。
